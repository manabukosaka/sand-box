# コントリビューション・ガイドライン (Contributing Guidelines)

本ドキュメントは、「Mini Datadog」プロジェクトにおける開発ワークフロー、ブランチ戦略、コーディング規約、および各種ガイドラインを定めたものです。
プロジェクトメンバー全員がこのルールに従うことで、コードの品質と開発スピードを高い水準で維持します。

---

## 1. ブランチ戦略 (Branching Strategy)

本プロジェクトでは、シンプルかつ高速にリリースサイクルを回すため、**GitHub Flow (または Trunk-Based Development)** を採用します。

*   **`main` ブランチ:**
    *   常にデプロイ可能（Deployable）で安定した状態を保ちます。
    *   直接コミットすることは原則禁止し、必ず Pull Request (PR) 経由でマージします。
*   **フィーチャーブランチ (Feature Branches):**
    *   新機能の追加、バグ修正、PoCなどはすべて `main` から分岐したフィーチャーブランチで作業します。
    *   **命名規則:** `<type>/<issue-or-short-desc>`
        *   例: `feature/log-ingestion-api` (新機能)
        *   例: `fix/memory-leak-buffer` (バグ修正)
        *   例: `docs/update-architecture` (ドキュメント更新)
        *   例: `poc/duckdb-write-perf` (技術検証)

---

## 2. コミットメッセージ規約 (Commit Conventions)

プロジェクトの履歴を追いやすくするため、**Conventional Commits** (規約に則ったコミットメッセージ) を採用します。

**フォーマット:**
```text
<type>(<scope>): <subject>

<body> (任意: 詳細な説明や理由)
```

*   **`<type>` の種類:**
    *   `feat`: 新機能の追加
    *   `fix`: バグ修正
    *   `docs`: ドキュメントのみの変更
    *   `style`: コードの動作に影響しないフォーマット変更（スペース、セミコロンなど）
    *   `refactor`: バグ修正や機能追加を含まないコードのリファクタリング
    *   `perf`: パフォーマンスを向上させるコード変更
    *   `test`: 不足していたテストの追加や既存テストの修正
    *   `chore`: ビルドプロセスや補助ツールの変更（プロダクションコードを含まない）
*   **`<scope>`:** (任意) 変更箇所（例: `api`, `ui`, `db`, `agent` など）。省略可。
*   **`<subject>`:** 変更内容の簡潔な要約（英語の命令形で記述）。

**例:**
*   `feat(api): add log ingestion endpoint`
*   `fix(ui): resolve chart rendering glitch on Firefox`
*   `docs: update ADR 0001 with PoC results`

## 3. コーディング規約と品質管理 (Coding Standards)

### 3.1 バックエンド (Rust)
Rustは強力な型システムとコンパイラを持っています。警告（Warnings）を無視せず、イディオマティックなコードを書くことを徹底します。

*   **フォーマッタ:** `cargo fmt` を使用します。CIでフォーマットチェックが走るため、コミット前に必ず実行してください。
*   **リンター:** `cargo clippy` を使用します。原則としてClippyの警告（Warnings）がゼロの状態を維持してください。
    *   やむを得ず警告を無視する場合は、必ずコード上に `#[allow(clippy::...)]` と共に、なぜ無視するのかコメントを残してください。
*   **テスト:** 重要なビジネスロジックやデータパース処理には、必ずユニットテスト（`#[test]`）を記述します。

### 3.2 フロントエンド (TypeScript / Next.js / React)
*   **型定義:** `any` の使用は原則禁止し、適切なインターフェースや型定義を行います。
*   **フォーマッタ:** `Prettier` を使用します。
*   **リンター:** `ESLint` (Next.js推奨設定) を使用します。
*   **コンポーネント設計:** プレゼンテーション層（見た目）とコンテナ層（ロジック・データフェッチ）を分離し、再利用性の高い設計を心がけます。

## 4. プルリクエスト (Pull Request) とレビュー

*   **PRの粒度:** 1つのPRは「1つの意味のある変更」に留め、レビューアーの負担を減らしてください。大きすぎるPRは分割を求められる場合があります。
*   **セルフレビュー:** PRを作成する前に、自分自身で差分（Diff）を確認し、不要なログ出力（`println!`, `console.log`）やコメントアウトされたコードが残っていないかチェックしてください。
*   **CI（継続的インテグレーション）:**
    *   PRを作成すると、自動的にビルド、テスト、リンター、フォーマッタのチェックが走ります（今後設定予定）。
    *   CIがパスしないPRはマージできません。
*   **レビュー承認:** 少なくとも1名以上の他の開発者から `Approve` を得た上で、`main` にマージします。

---

## 5. ドキュメンテーション (Docs-as-Code)

*   アーキテクチャの変更や主要な技術選定（DBの変更、通信プロトコルの決定など）は、コードを書き始める前に必ず **ADR (Architecture Decision Record)** として `docs/adr/` に記録し、レビューを受けてください。
*   README.md や 各種設計ドキュメントは常に最新の状態を保ち、「ドキュメントが嘘をついている」状態を作らないようにしてください。

---

## 6. 各開発工程のガイドライン (Development Phase Guidelines)

本プロジェクトの開発プロセス（`docs/project_plan.md` 参照）に沿った、各工程ごとの具体的な作業ガイドラインです。

### 6.1 Phase 1: 構想・アーキテクチャ設計 (Inception & Design)
*   **ドキュメント・ファースト:** 実装に入る前に、要件定義 (`requirements.md`)、アーキテクチャ (`architecture.md`)、APIスキーマ (`api_schema.md`) を作成・更新すること。
*   **ADRの作成:** 主要な技術選定（データベース、通信プロトコル、フレームワークなど）を行う場合は、必ず `docs/adr/` に Architecture Decision Record (ADR) を作成し、選定理由と代替案の比較を残すこと。
*   **レビュー:** 設計ドキュメントの変更はPRとして提出し、チームメンバーの合意（Approve）を得ること。

### 6.2 Phase 2: 技術検証 (Proof of Concept - PoC)
*   **スコープの限定:** PoCは「最もリスクの高い技術的課題の検証」に特化し、UIの見た目やエッジケースの処理には時間をかけないこと。
*   **隔離された環境:** PoCのコードは `poc/` ディレクトリ配下に作成し、本番コード (`server/` や `web/`) には混入させないこと。
*   **検証レポート:** 検証結果（パフォーマンス計測データなど）は `docs/poc_report.md` または関連するADRに追記し、次のフェーズの意思決定の材料とすること。

### 6.3 Phase 3: MVP開発 (MVP Development)
*   **コア機能への集中:** ログ・メトリクス受信、保存、検索、可視化という「Minimum Viable Product」の必須機能のみを実装すること。Nice-to-haveな機能はIssueとして起票し、後回しにすること。
*   **モックの活用:** フロントエンドとバックエンドを並行開発するため、APIスキーマに基づいたモック（ダミーデータ）を積極的に活用すること。
*   **早期結合テスト:** 単体テストだけでなく、早い段階でバックエンドとフロントエンドを結合し、エンドツーエンドでの動作確認を行うこと。

### 6.4 Phase 4: アルファリリースとフィードバックループ (Alpha Release)
*   **Dogfooding:** 開発者自身の環境でMini Datadogを実際の監視ツールとして運用（Dogfooding）し、生のフィードバックを得ること。
*   **メトリクスの収集:** アプリケーション自体のパフォーマンス（CPU/メモリ使用量、レスポンスタイム）を測定し、ボトルネックを特定すること。
*   **バグと改善の切り分け:** 発見された課題は、致命的なバグ（即時修正）とUI/UXの改善要望（バックログ行き）に分類してIssue管理すること。

### 6.5 Phase 5: 機能拡充と安定化 (Enhancement & Stabilization)
*   **テストの拡充:** ユニットテストのカバレッジを向上させ、主要な機能に対するインテグレーションテストを整備すること。
*   **リファクタリング:** MVPフェーズで妥協したコードの技術的負債を返済すること。
*   **ユーザードキュメント:** インストール手順、APIリファレンス、トラブルシューティングガイドなどのユーザー向けドキュメントを充実させること。
