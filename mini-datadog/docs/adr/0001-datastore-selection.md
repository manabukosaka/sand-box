# ADR 0001: データストアの選定 (Data Store Selection)

## 1. コンテキストと課題 (Context & Problem Statement)
「Mini Datadog」は、中小規模環境向けの軽量な監視プラットフォームである。毎秒数千〜数万件のログおよびメトリクスデータを受信（Write-heavy）し、ダッシュボードから高速に検索・集計（Read-heavy）できる必要がある。
同時に、「単一バイナリまたはシンプルなDocker Compose構成でのセルフホスト容易性」という強い制約があるため、外部の重厚なデータベース（Elasticsearch, Prometheus + Thanos等）への依存は避けたい。
本ドキュメントでは、ソフトウェア品質特性、アーキテクチャ特性、およびOSSとしての持続性（ライセンス・コミュニティ）の観点から意思決定分析（DA）を行い、最適なデータストアを選定する。

## 2. 評価基準 (Decision Criteria)
1. **OSS持続性・リスク (Sustainability & Risk)**
   - ライセンスの安全性（パーミッシブ、将来の変更リスク低、商用利用/セルフホストに制限なし）
   - コミュニティの活発さとメンテナンス継続性（特定の1社による支配的ガバナンスリスクの排除）
2. **ソフトウェア品質特性 (Software Quality Characteristics - ISO/IEC 25010)**
   - **性能効率性 (Performance):** 高スループットの書き込み、低レイテンシの検索・集計
   - **運用性 (Operability):** 単一バイナリ化への組み込み容易性、運用（バックアップ、データパージ等）のシンプルさ
   - **信頼性 (Reliability):** 障害時のデータ保護、安定したメモリ/ストレージ使用量
3. **アーキテクチャ特性 (Architectural Characteristics)**
   - **シンプルさ (Simplicity):** アーキテクチャを複雑にしない。開発・学習コストの抑制
   - **親和性 (Rust Integration):** Rustエコシステムとの統合のしやすさ、安全性

## 3. 比較候補 (Alternatives)

### A. SQLite (with WAL mode)
- **概要:** 世界で最も使われている組み込みRDBMS。
- **ライセンス:** Public Domain (リスク極小)。
- **特徴:** Rust (rusqlite / sqlx) との親和性が極めて高い。WAL (Write-Ahead Logging) モードにより並行読み書き性能が向上する。時系列データのフルスキャンには弱いが、適切なインデックスと時間単位のパーティショニング（ファイル分割）で一定のカバーが可能。

### B. DuckDB
- **概要:** 分析向け（OLAP）の組み込みRDBMS。列指向（Columnar）アーキテクチャ。
- **ライセンス:** MIT License (安全)。
- **特徴:** 大量のログやメトリクスの集計・検索（GROUP BY, フィルタリング）が極めて高速。Rustバインディングあり。ただし、毎秒数万件の細かいINSERT（OLTP的アクセス）には弱いため、アプリケーション側でのインメモリバッチ化（Appenderの利用）が必須となる。

### C. RocksDB (または Sled)
- **概要:** 高速なKey-Value Store (LSM Treeベース)。
- **ライセンス:** Apache 2.0 / GPL (RocksDB), MIT (Sled)。
- **特徴:** 書き込み性能は圧倒的。しかし、KVSであるため検索や集計のためのインデックス・クエリエンジンを自前で実装する必要があり、開発コスト（シンプルさの欠如）が極めて高い。要件の「検索・可視化」を満たすための車輪の再発明が発生する。

### D. VictoriaMetrics (Single Node)
- **概要:** 高パフォーマンスな時系列データベース。
- **ライセンス:** Apache 2.0。
- **特徴:** Prometheus互換で、メトリクスの保存・圧縮・クエリに特化（ログも系列製品のVictoriaLogsで対応可能）。性能は申し分ないが、Rustプロセスとは別プロセスとして動かす必要があり、「単一バイナリ」要件から外れる。Docker Compose構成が前提となる。

## 5. 推奨と決定 (Recommendation)

**推奨案: B. DuckDB （メインの分析・保存エンジンとして）**
※ または開発初期のMVPフェーズとしては **A. SQLite**

**選定理由:**
Mini Datadogの最大の提供価値は「ダッシュボードでの高速な分析・可視化」と「運用（セルフホスト）の容易さ」にあります。
- ログのキーワード検索やメトリクスの時系列集計（時間枠での平均化など）は典型的なOLAPワークロードであり、**DuckDB** がソフトウェア品質特性（検索パフォーマンス）として最も適しています。ライセンス（MIT）や単一バイナリへの組み込み容易性も要件を満たします。
- アーキテクチャ上の課題として「細かい大量のINSERTへの弱さ」がありますが、これはRustバックエンド側でインメモリのリングバッファやチャネルを用いて数秒〜数十秒分の中間データをバッファリングし、DuckDBへ一括書き込み（バルクインサート）を行う設計で解決可能です。
- 万が一、初期の開発スピード（シンプルさ）を最優先する場合は、学習コストがゼロに近い **SQLite (WALモード)** を採用し、データ量が増えて検索パフォーマンスの限界が見えた段階でDuckDBへ移行するという戦略も妥当です。

## 6. 影響と次のアクション (Consequences & Next Actions)

- **決定事項:** [ ] 未決定 (プロジェクトオーナーの判断待ち)
- **今後のアクション:**
  1. **PoC (Proof of Concept) の実施:** Rustを用いて、インメモリバッファ + DuckDB（またはSQLite）への一括書き込みを行う小規模な検証コードを実装する。
  2. **負荷テスト:** ターゲットとなる最低限のハードウェア（例: 2vCPU, 4GB RAM）で、毎秒1万件のWriteと並行した集計Readが成立するかを検証し、リソース消費（CPU/メモリ）を計測する。
  3. PoCの結果をもって、採用技術を最終決定し、本ADRを「Approved」ステータスに更新する。
