# ADR 0002: リアルタイム通信方式の選定 (Real-time Communication Method)

## 1. コンテキストと課題 (Context & Problem Statement)
「Mini Datadog」の主要機能の一つである「Live Tail」では、サーバーが受信したログをほぼリアルタイム（サブ秒のレイテンシ）でブラウザ上の Web UI へ転送する必要がある。
このリアルタイム通信を実現するための技術スタック（SSE または WebSocket）を選定する。

## 2. 評価基準 (Decision Criteria)
1. **シンプルさ (Simplicity):** 実装と運用の容易さ。特にバックエンド（Rust）とフロントエンド（React）でのライブラリ支援状況。
2. **パフォーマンスと効率 (Efficiency):** サーバーのリソース消費（メモリ、接続維持コスト）と通信オーバーヘッド。
3. **通信要件 (Communication Pattern):** 単方向（Server -> Client）で十分か、双方向（Bidirectional）が必要か。
4. **インフラ親和性 (Infrastructure Compatibility):** プロキシ（Nginx, LB）や HTTP/2 経由での動作安定性。

## 3. 比較候補 (Alternatives)

### A. SSE (Server-Sent Events)
- **概要:** 標準的な HTTP 接続を維持し、サーバーからクライアントへデータをプッシュし続ける技術。
- **メリット:**
  - HTTP ベースであるため、標準的な HTTP ツールやプロキシとの親和性が高い。
  - 実装が極めてシンプル（特別なプロトコルアップグレードが不要）。
  - 自動再接続機能がブラウザ標準で組み込まれている。
  - 単方向通信（サーバーからログを流すだけ）という Live Tail の要件に合致している。
- **デメリット:**
  - 単方向のみ（クライアントからサーバーへの送信は別途通常の HTTP リクエストが必要）。
  - HTTP/1.1 ではブラウザごとの接続数制限（通常6接続）に抵触しやすい（HTTP/2 では解消）。

### B. WebSockets
- **概要:** TCP 上で双方向通信を行うためのプロトコル。
- **メリット:**
  - 双方向通信が可能。将来的にクライアントからの操作（ストリームの動的変更など）を頻繁に行う場合に有利。
  - バイナリデータの送受信も可能。
- **デメリット:**
  - 独自のプロトコル（`ws://`）であるため、プロキシやロードバランサーの設定が必要になる場合がある。
  - 再接続ロジックを自前で実装する必要がある。
  - ステートフルな接続管理のコストが SSE よりわずかに高い。

## 4. 意思決定 (Decision)

**採用案: A. SSE (Server-Sent Events)**

**選定理由:**
1. **要件との合致:** Live Tail は本質的に「サーバーからクライアントへの単方向のログ流し込み」であり、SSE で完全に要件を満たせる。
2. **実装のシンプルさ:** Rust (Axum / Tokyo) における SSE の実装は非常に軽量であり、フロントエンド（`EventSource` API）も標準でサポートされている。
3. **HTTP/2 への期待:** 単一バイナリで HTTP/2 サーバーとして動作させる場合、SSE の接続数制限問題も回避可能であり、最も軽量な選択肢となる。
4. **再接続の容易さ:** ネットワーク瞬断時の再接続がブラウザ標準機能として提供されているため、フロントエンドのコードを簡素化できる。

## 5. 影響と次のアクション (Consequences & Next Actions)

- **影響:**
  - ストリームのフィルタリング条件変更などは、クエリパラメータで制御するか、一度切断して新しい URL で再接続する方式をとる。
- **今後のアクション:**
  - PoC において、Rust の Axum フレームワークを用いた SSE のサンプル実装を行い、並行接続時のメモリ消費量を確認する。
