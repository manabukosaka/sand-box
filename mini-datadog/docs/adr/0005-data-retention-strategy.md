# ADR 0005: データクリーンアップジョブと保持期間戦略 (Data Cleanup Job and Retention Strategy)

## 1. ステータス (Status)
Accepted

## 2. コンテキスト (Context)
mini-datadog では、継続的にログやメトリクスが蓄積されるため、ストレージ容量の圧迫を防ぐために古いデータを定期的に削除（パージ）するデータクリーンアップジョブが必要である。
当初の要件では「30日経過したデータのパージ」が想定されているが、データストアとして採用した **DuckDB** には特有のアーキテクチャ上の制約があるため、以下の点を考慮した慎重な設計が求められる。

1. **DuckDBのシングルライター制約**: 書き込みトランザクション（DELETE含む）が実行中の間、データベースファイル全体がロックされ、他の書き込み（インジェストAPIからの新規ログ追加など）がブロックされる。
2. **パフォーマンスへの影響**: 単一トランザクションで大量のデータを `DELETE` すると、ロック時間が長時間化しインジェスト処理がタイムアウトする危険性がある。また、データの断片化（フラグメンテーション）が発生し、後続の検索レイテンシに悪影響を及ぼす可能性がある。
3. **設定の柔軟性**: 保持期間を「30日固定」にするか、運用環境に応じて変更可能にするかの判断が必要である。

## 3. 決定事項 (Decision)

シニアエンジニアが実装に入るための「正」の設計として、以下の戦略を採用する。

### 3.1. クリーンアップ実行戦略: 「低負荷時間帯のバッチサイズ分割DELETE」
- **実行タイミング:** システムのトラフィックが最も低いと想定される時間帯（例: 毎日午前3:00）に、バックグラウンドの非同期タスクとして実行する。
- **バッチサイズ分割:** 削除対象の期間（例: 30日以上前のデータ）を一括で `DELETE` するのではなく、トランザクションを小さく分割する（例: 1日分、あるいは数時間分ずつ `DELETE` を発行する）。
- **ロック解放のためのインターバル:** 各分割 `DELETE` トランザクションの間に意図的な非同期スリープ（例: 数十〜数百ミリ秒）を挟む。これにより、APIサーバー側がインジェスト用の書き込みロックを取得する「隙間」を作り、データの取りこぼしを防ぐ。
- **WALの整理:** 一連の削除処理が完了した後、明示的に `CHECKPOINT;` コマンドを発行し、Write-Ahead Log (WAL) の肥大化を防ぎディスクに確実な反映を行う。

### 3.2. 設定の柔軟性: 「環境変数による動的設定」
- 保持期間はハードコードせず、環境変数 `DATA_RETENTION_DAYS` によって動的に変更可能とする。
- デフォルト値は `30`（日）とする。
- 理由: ユーザーのストレージ容量や要件に応じて、再コンパイルなしで保持期間を柔軟に変更（例: 7日間に短縮）できるようにするため。

## 4. 決定の根拠 (Rationale)
- **なぜ分割DELETEなのか:** シングルライター制約下において長時間の書き込みロックは致命的である。バッチ分割とインターバルの導入によって、クリーンアップジョブが裏で動いていても、メインのデータ収集（書き込み）が実質的にブロックされないように制御できる。
- **なぜ動的設定なのか:** ソフトウェアとしての汎用性を高めるため。環境変数の読み込みを追加する実装コストは極めて低く、対費用効果が高い。
- **パフォーマンスの保護:** `CHECKPOINT` の実行により、不要な領域が適切にマークされ、ファイルの無駄な肥大化や検索パフォーマンスの劣化を抑えられる。

## 5. 棄却した代替案 (Alternatives Considered)

- **代替案A: `CREATE TABLE AS SELECT` によるテーブル再構築（スワップ）**
  - **内容:** 残すべきデータ（過去30日分）のみを新しいテーブルにコピーし、古いテーブルを `DROP` してリネームする手法。
  - **棄却理由:** DuckDB において一括の大量削除に対するベストプラクティスではあるものの、データのコピー中に発生する「新規ログの取り扱い（差分同期）」が複雑化し、MVP フェーズの実装コストに見合わないため今回は見送った。
- **代替案B: 時間ベースのパーティショニング（複数DBファイル化）**
  - **内容:** 日別に別々の DuckDB ファイルを作成し、30日前のファイルを単純にファイルシステム上から削除する手法。
  - **棄却理由:** 検索時に複数ファイルを横断してクエリする機能（`ATTACH`等）の管理が必要となり、現在の単一DBファイル設計（`db.rs`）から大きく乖離するため、将来の拡張課題として保留した。
- **代替案C: 保持期間のハードコード**
  - **棄却理由:** 運用環境ごとのディスク容量の違いを吸収できず、スケールアップを強制することになるため不適切と判断した。

## 6. 結果 (Consequences)
- シニアエンジニアは本ADRに基づき、`server` クレート内にバックグラウンドタスク（例: `tokio::spawn` を利用した定期実行タイマー）を実装する。
- クリーンアップ対象は `logs` テーブルおよび `metrics` テーブルの `timestamp` カラム基準となる。
- `DATA_RETENTION_DAYS` の環境変数読み込み処理を追加し、起動時にパースする仕組みを構築する。
