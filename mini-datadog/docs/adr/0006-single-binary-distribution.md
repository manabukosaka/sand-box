# ADR 0006: シングルバイナリ配信アーキテクチャの採用 (Single Binary Distribution Architecture)

## 1. ステータス (Status)
Proposed

## 2. コンテキスト (Context)
Mini Datadog はローカル環境や単一サーバーで容易にデプロイ・起動できることが求められている。現在、フロントエンドは Next.js、バックエンドは Rust (Axum) で別々に構築されているが、ユーザーがそれぞれを個別に起動・管理するのは運用上の負担（DevOps的負荷）となる。
したがって、運用を極めてシンプルにするために、バックエンドとフロントエンドを1つの実行可能バイナリとして配布する「シングルバイナリ」アプローチが必要である。

## 3. 決定事項 (Decision)
`rust-embed` クレートを利用して Next.js でビルドされた静的アセットを Rust のバックエンドバイナリに直接埋め込み、Axum ルーター経由で配信する構成を採用する。

### 3.1 配信方式
- Next.js を `output: 'export'` モードでビルドし、完全な静的サイト (HTML/CSS/JS) を生成する。
- Rust 側で `rust-embed` を使用し、生成された `web/out` ディレクトリの内容をコンパイル時にバイナリに埋め込む。
- Axum にて、`/` や `/assets/*`、その他静的ファイル（`.js`, `.css` など）に対するリクエストを処理する専用のハンドラを設ける。

### 3.2 ルーティングの整合性 (SPA フォールバック)
Next.js によるクライアントサイドルーティングと Axum のサーバーサイドルーティングの競合を防ぐため、以下のルーター設定を実装する。
1. **API ルート**: `/api/*` など、バックエンド専用のエンドポイントを最優先で評価する。
2. **静的ファイル**: リクエストパスが `rust-embed` に埋め込まれたファイル名（例: `/_next/static/*`, `/favicon.ico` 等）と一致する場合、そのファイルを適切な MIME タイプと共に返す。
3. **フォールバック**: 上記に該当しないすべての GET リクエスト（例: `/search`, `/dashboards` への直接アクセス）は、Next.js のクライアントルーターが処理できるように、ルートの `index.html` を返す（SPAフォールバック）。

### 3.3 ビルドプロセス
Rust のコンパイル前にフロントエンドのアセットが確実にビルドされていることを保証するため、ビルドパイプラインを整備する。
- 開発環境および CI/CD パイプラインにおいて、`Makefile` やタスクランナーを使用し、必ず `cd web && npm run build` が成功した後に `cargo build` を実行する依存関係を定義する。
- （オプションとして）Rust の `build.rs` を利用し、`cargo build` 実行時に `web/out` の存在確認や、必要に応じた `npm run build` の自動実行フックを追加することで、開発者のビルド体験を向上させる。

## 4. 代替案とその棄却理由 (Alternatives Considered)

- **代替案1: Docker Compose による複数コンテナ構成**
  - **内容**: Nginx 等のリバースプロキシ、Node.js サーバー、Rust バックエンドをコンテナとして分離する。
  - **棄却理由**: 運用コストが増大し、ユーザーが環境構築の手間（Docker の要求など）を負うことになる。ローカルでの「サクッと起動」という要件に反する。
- **代替案2: 実行時の静的ファイル配信 (動的読み込み)**
  - **内容**: Axum の `tower-http` の `ServeDir` を用いて、実行時のファイルシステム上の `web/out` ディレクトリから配信する。
  - **棄却理由**: 実行時にディレクトリパスの管理が必要になり、配布物が「バイナリファイル1つ」にならないため棄却。

## 5. 結果 (Consequences)

### ポジティブ
- ユーザーは `mini-datadog` のバイナリを1つダウンロードして実行するだけで、Web UI もバックエンドも利用可能になる（究極のデプロイ容易性）。
- Docker や Node.js のランタイムに依存しないため、多様な環境に配布しやすい。

### ネガティブ・考慮事項
- **ビルド時間の増加**: フロントエンドのビルド時間が Rust のビルド全体の時間に加算される。
- **バイナリサイズの増加**: フロントエンドのアセットが直接バイナリに埋め込まれるため、最終的な実行ファイルのサイズが肥大化する。
- **開発時のワークフロー**: バックエンドのみを変更した場合でも、フロントエンドの組み込みロジックとの競合を避けるために、開発中は個別にプロセスを起動する構成（Next.js の `npm run dev` と Rust API の分離）も併せて維持する必要がある（CORS設定などの考慮が必要）。
