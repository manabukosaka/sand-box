# ADR (Architecture Decision Records) 運用ガイドライン

## 1. ADR の目的
プロジェクトにおける重要な技術的・設計上の意思決定（およびその背景、代替案、トレードオフ）を記録し、将来のプロジェクト参加者が「なぜ現在の設計になったのか」を理解できるようにします。

## 2. ADR の作成・更新ルール

### 2.1 変更禁止の原則 (Immutable ADRs)
一度作成され、採番（例: `0001-*.md`）された ADR ファイルは、**原則として後から内容を編集してはいけません。**
これにより、意思決定の履歴を不変のタイムラインとして残します。

### 2.2 ステータスの更新と変更の記録
意思決定に変更があった場合や、ステータス（例: 提案中 -> 承認済み）を更新する場合は、以下のいずれかの方法をとります。

1.  **新しい ADR の作成 (推奨):**
    新しい番号で ADR を作成し、その中で以前の ADR を参照（例: `Supersedes 0001`）します。
2.  **追記用ファイル / メタデータファイル:**
    ADR の本体ではなく、ステータスや検証結果を記録するための別ファイル、または ADR 内の「ステータス」セクションを更新するための特別な「変更用 ADR」を発行します。

### 2.3 ファイル命名規則
`NNNN-short-description.md` (NNNN は 4 桁の連番)

## 3. ADR の構造
1.  **タイトル (Title):** 簡潔な要約
2.  **ステータス (Status):** Proposed (提案中), Accepted (承認), Superseded (代替済み), Deprecated (廃止)
3.  **コンテキスト (Context):** 課題の背景と制約
4.  **決定事項 (Decision):** 採用する解決策
5.  **結果 (Consequences):** 決定によるメリット・デメリット
6.  **参照 (References):** 関連する ADR やドキュメント
