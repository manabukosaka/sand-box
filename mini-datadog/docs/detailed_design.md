# 詳細設計書 (Detailed Design)

## 1. データベース設計 (Database Schema - DuckDB)

DuckDB を使用した、ログおよびメトリクス保存用のスキーマを定義します。

### 1.1 ログテーブル (`logs`)
ログデータは、検索と集計の効率を最大化するため、以下のカラム構成とします。

| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `timestamp` | `TIMESTAMP` | エージェント側でのログ発生日時 |
| `received_at` | `TIMESTAMP` | サーバー側での受信日時 (パーティショニング/バリデーション用) |
| `level` | `VARCHAR` | ログレベル (info, error, etc.) |
| `service` | `VARCHAR` | 送信元サービス名 |
| `message` | `TEXT` | ログメッセージ本文 |
| `tags` | `JSON` | 任意のタグ情報 (env, region 等) |
| `attributes` | `JSON` | 構造化された属性データ |

**インデックス/最適化:**
- `received_at` を主軸としたソート済みテーブルとして保持し、異常な `timestamp` に対する耐性を確保。
- `service` および `level` カラムに対するフィルタリングの高速化。

### 1.2 メトリクステーブル (`metrics`)
| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `timestamp` | `TIMESTAMP` | 計測日時 |
| `name` | `VARCHAR` | メトリクス名 |
| `value` | `DOUBLE` | 計測値 |
| `service` | `VARCHAR` | 送信元サービス名 |
| `tags` | `JSON` | タグ情報 |

### 1.3 認証テーブル (`api_keys`)
将来の拡張性を考慮し、APIキーを管理するテーブルを用意します（初期はメモリキャッシュ併用）。
| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `key_hash` | `VARCHAR` | APIキーのハッシュ値 (主キー) |
| `service_name` | `VARCHAR` | 紐付けられたサービス名 |
| `created_at` | `TIMESTAMP` | 作成日時 |

## 2. インメモリバッファと永続化処理

### 2.1 バックプレッシャーと動的制御
- **バッファ制限:** デフォルトのチャネル容量を 10,000 件（約10MB想定）に引き下げ、軽量環境での OOM リスクを低減する。この値は環境変数 `BUFFER_SIZE` で調整可能とする。
- **バックプレッシャー連動:** 書き込みワーカーが詰まり、チャネルがフルになった場合、Ingestion API は即座に `429 Too Many Requests` を返却する。
- **動的フラッシュ:** 1,000件到達または1秒経過で実行。書き込み処理自体の所要時間をメトリクスとして計測し、DB パフォーマンスの劣化を早期検知可能にする。

### 2.2 自己監視メトリクス
サーバー自身の健全性を監視するため、以下の内部メトリクスを自動的に収集する。
- `internal.buffer.usage`: 現在のバッファ蓄積件数
- `internal.db.write_latency_ms`: DuckDB へのバルクインサート所要時間
- `internal.ingest.error_count`: 認証失敗やバリデーションエラーの回数

### 2.2 データ損失対策とWAL
- **MVP:** プロセス異常終了時のバッファ内データ消失を許容する。
- **Next Step:** 高信頼性が求められるユースケース向けに、DuckDB への書き込み前にローカルディスクの軽量なログファイル（WAL）へ追記するオプションの実装を計画。

## 3. 認証・認可 (Security)

### 3.1 APIキー認証の実装
- Axum の `FromRequest` または `Middleware` で `X-API-Key` ヘッダーを検証。
- 起動時に `api_keys` テーブルから有効なキーを `DashMap` (Thread-safe Map) 等にキャッシュし、リクエストごとの DB 問い合わせを回避する。

---

## 4. フロントエンド・バックエンド統合

### 4.1 静的アセットの埋め込み
- `rust-embed` クレートを使用。
- `dist/` ディレクトリ（Next.js のビルド結果）をバイナリに含め、Axum のルーティングで `get("/assets/*")` 等として配信。

---

## 5. 実装優先順位 (Phase 3 開発順序)
1. **基盤:** DuckDB 接続管理とテーブル作成ロジック。
2. **受信:** `Ingestion API` と `mpsc` バッファの実装。
3. **認証:** APIキー検証ミドルウェア。
4. **検索:** `Query API` の SQL 生成ロジック。
5. **UI:** ログ検索画面と Live Tail 画面のプロトタイプ。
