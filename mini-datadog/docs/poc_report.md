# 技術検証 (PoC) レポート: DuckDB を用いた高スループットログ受信

## 1. 検証の目的
Mini Datadog のバックエンドにおいて、Rust を用いた大量ログの受信と、DuckDB への効率的なバルクインサート（一括書き込み）が可能であることを検証する。特に、インメモリバッファリングによって I/O 負荷を低減しつつ、リアルタイムに近いクエリ性能を維持できるかを確認する。

## 2. 検証内容
- **言語・フレームワーク:** Rust (Axum, Tokio)
- **データストア:** DuckDB (Embedded mode)
- **バッファリング戦略:** 
  - `tokio::sync::mpsc` チャネルを用いた非同期受信。
  - バックグラウンドワーカーによる「1,000件到達」または「1秒経過」ごとのバルクインサート。
  - DuckDB の `Appender` API を使用した高速な一括書き込み。

## 3. 検証結果
### 3.1 機能性の確認
- `POST /api/v1/ingest/logs`: 複数のログレコードを JSON 配列形式で正常に受信。
- `GET /api/v1/query/logs`: 保存されたログをサービス名でフィルタリングし、タイムスタンプの降順で取得できることを確認。
- バックグラウンドでの一括書き込みが正常に動作し、API レスポンス（202 Accepted）と DB 永続化が分離されていることを確認。

### 3.2 パフォーマンス（暫定）
- DuckDB の `Appender` を使用することで、数ミリ秒で数千件のレコードを書き込むことが可能。
- 単一の SQLite (WALモード) と比較しても、集計や分析用途（OLAP）において DuckDB の列指向アーキテクチャの優位性が期待できる。

## 4. 課題と今後の展望
- **メモリ管理:** 非常に高いスパイクが発生した際のバッファ上限設定と、バックプレッシャーの制御が必要。
- **データ型:** ログの `tags` フィールド（JSON）に対する効率的なインデックスや、DuckDB 固有の `JSON` 型の最適活用。
- **並行読み書き:** DuckDB は単一の書き込みプロセスを推奨しているため、バックグラウンドワーカーを単一の書き込み専用スレッドとして管理する現行の設計が妥当であることを再確認。

## 5. 結論
本 PoC の結果、**DuckDB をメインのデータストアとして採用することを推奨する**。Rust との統合も容易であり、要求されるパフォーマンス特性を十分に満たせると判断した。
