# ドキュメント品質基準 (Documentation Style Guide)

本ガイドラインは、プロジェクト内のすべてのドキュメント（技術ドキュメント、ユーザーマニュアル、API仕様書など）を世界最高水準に保つための基準です。Documentation-First Policyに基づき、コードと同等以上の品質でドキュメントをメンテナンスしてください。

## 1. 構成の標準化 (Standardized Structure)
すべてのドキュメントは、以下の構成を基本とします。
- **Title (タイトル):** ドキュメントの目的が一目でわかる簡潔なタイトル。
- **Overview (概要):** このドキュメントが何について書かれているか、対象読者は誰かを明記する（3〜5行程度）。
- **Prerequisites (前提条件):** このドキュメントを読む、または実行するために必要な事前知識や環境。
- **Body (本文):** 論理的な見出し（H2, H3）を用いて階層化し、結論から先に書く（PREP法を意識する）。
- **References (参考資料):** 関連するADR、外部リンク、他ドキュメントへのリンク。

## 2. 用語の統一 (Terminology Consistency)
- プロジェクト内で使用する専門用語、機能名、アーキテクチャ要素の名称は必ず統一すること。
- 表記揺れ（例: 「ユーザ」と「ユーザー」、「サーバ」と「サーバー」）を防ぐため、JIS規格または一般的な技術文書の慣例（長音符をつける等）に従うこと。
- 英語の頭字語（API, DB, UIなど）は大文字で統一する。

## 3. 図解の推奨 (Visual Documentation)
- 複雑なアーキテクチャ、システム間の連携、状態遷移（ステートマシン）を説明する場合は、テキストだけでなく必ず図解を含めること。
- Mermaid.js を標準の作図ツールとして使用し、Markdown内にコードとして埋め込むことで、バージョン管理と継続的な更新を容易にする。

## 4. 明確さと簡潔さ (Clarity and Conciseness)
- **能動態の使用:** 可能な限り能動態を使用し、「誰が（何が）」「何をするか」を明確にする。
- **箇条書きの活用:** 手順や条件の羅列は、箇条書き（リスト）を用いて視認性を高める。
- **コード例の提供:** APIの使い方や設定ファイルの説明には、必ず実用的なコードスニペット（シンタックスハイライト付き）を含めること。

## 5. 継続的更新 (Continuous Maintenance)
- ドキュメントは「生きた情報」である。システムの変更（機能追加、仕様変更、非推奨化）が行われた際は、対応するドキュメントも同一のPR内で必ず更新する（Documentation-First Policy）。
