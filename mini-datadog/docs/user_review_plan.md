# Mini Datadog アルファ版 ユーザーレビュー計画 (User Acceptance Testing Plan)

## 1. レビューの目的 (Objectives)
本ユーザーレビュー（UAT）の主目的は、Mini Datadog アルファ版リリースに向け、以下の観点からプロダクトの価値と課題を検証することです。
- **プロダクトの実用性 (Utility):** 提供される機能が、実際の運用やトラブルシューティングの現場において真に役立つか。
- **UXの直感性 (Intuitive UX):** ユーザーが迷うことなく、目的の操作（ログの検索、Live Tailの閲覧など）をスムーズに実行できるか。
- **ドキュメントの完全性 (Documentation Completeness):** READMEやユーザーガイド等のドキュメントのみで、ユーザーが自立してセットアップや操作を行えるか。

## 2. 評価ペルソナ (Personas)
検証の精度を高めるため、以下の2つの代表的なペルソナを設定し、それぞれの視点から評価を行います。

### Persona A: インフラエンジニア (SRE)
- **役割:** システムの安定稼働、リソース監視、インフラのプロビジョニングを担当。
- **評価フォーカス:** 初期セットアップの容易さ、単一バイナリによるデプロイ体験、リソース管理の妥当性。
- **前提スキル:** Linuxサーバーの操作、コマンドラインツールの利用に長けている。

### Persona B: アプリケーション開発者 (Dev)
- **役割:** アプリケーションの機能開発、バグ修正、障害発生時のトラブルシューティングを担当。
- **評価フォーカス:** ログ検索画面（Logs Search）の使い勝手、Live Tailによるリアルタイム監視の実用性、エラー特定までのスピード。
- **前提スキル:** ログ分析ツール（Datadog等）の基本的な利用経験がある。

## 3. テストシナリオ（タスク）
各ペルソナに対して、以下の具体的なシナリオ（タスク）を提示し、実際の環境で操作を実施してもらいます。

### シナリオ1 (SRE向け)
**タスク:** 「提供された README とユーザーガイドのみを参照し、Mini Datadogの単一バイナリをサーバー上で起動して初期設定を完了せよ。」
- **ゴール:** サーバーが正常に起動し、Web UIにアクセスできる状態になること。
- **検証ポイント:** ドキュメントの記載に不足や誤りがないか。単一バイナリの配布形態がデプロイの障壁を下げているか。

### シナリオ2 (Dev向け)
**タスク:** 「Live Tail 画面を開き、リアルタイムに流れるログを監視せよ。その後、特定のテキスト（例: 'Connection timeout'）を含むエラーログを発見し、そのログ行をコピーせよ。」
- **ゴール:** Live Tailがスムーズに動作し、目当てのログを視認・取得できること。
- **検証ポイント:** 大量に流れるログの中で、スクロールの一時停止やフィルタリングが直感的に行えるか。

### シナリオ3 (Dev向け)
**タスク:** 「Logs Search 画面にアクセスし、過去24時間以内に発生した 'ERROR' レベルのログを抽出せよ。抽出した結果から、最も新しいエラーのタイムスタンプを特定せよ。」
- **ゴール:** 正しいクエリまたはUI操作でログを絞り込み、該当のタイムスタンプを特定できること。
- **検証ポイント:** 検索UIの使いやすさ、時間の絞り込み操作の直感性、検索結果の表示速度と視認性。

## 4. 評価指標 (Metrics)
各テストシナリオの実施において、以下の指標を用いて定量・定性の両面から評価を記録します。

- **タスク完了までの時間 (Time on Task):**
  - 各シナリオの開始から完了までにかかった実時間を測定します。操作の効率性を測るための定量的な指標となります。
- **操作の迷い (Friction Points):**
  - ユーザーがUI上で立ち止まった箇所、間違ったボタンを押した回数、あるいは「どうすればいいか分からない」と発言した回数を記録します。UXの改善点を洗い出します。
- **ドキュメントの不足点 (Documentation Gaps):**
  - ドキュメントを読んでも解決しなかった疑問点、説明が不十分でユーザーが推測に頼らざるを得なかった箇所を特定し、マニュアルの改訂に繋げます。
